# 省钱购物助手

商品转链 + 跨平台比价 + 历史价格走势，帮用户找到最划算的购买方式。

## 术语约定

全文统一使用以下术语，禁止混用：

| 术语 | 含义 | 对应字段 |
|------|------|----------|
| 到手价 | 用户实际支付的最终价格（含券后） | convert 接口的 `finalPrice` |
| 原价 | 商品标价（未优惠） | convert 接口的 `itemPrice` |
| 历史最低价 | 该商品有记录以来的最低售价 | history-price 接口的 `historyLowestPrice` |

## 路由决策

根据用户意图选择接口，所有接口通过 `bash {baseDir}/scripts/run.sh call <接口名>` 调用：

| 用户意图 | 调用接口 | 参数 |
|----------|----------|------|
| 只发了商品链接/淘口令，没说别的 | `convert` + `compare-price` + `history-price`（全套） | 见下方 |
| 明确说"转链"、"优惠券" | `convert` | `--tpwd <链接>` |
| 明确说"比价"、"哪家便宜"、"最低价" | `compare-price` | `--productIdentifier <链接>` |
| 明确说"历史价"、"价格走势" | `convert`（取到手价） + `history-price` | `--productIdentifier <链接>` |
| "值不值得买"、"该不该入手" | `convert` + `history-price` + `compare-price`（全套） | 见下方 |
| 同时问多个 | 分别调用对应接口 | — |

**关键规则**：
- **用户只发链接没说意图时，默认走全套**——收到链接后立刻先输出一句状态提示，然后调用第一个接口，拿到结果立刻输出卡片，再调下一个（详见「全套流程」章节）。**禁止等所有接口调完再统一输出。**
- 历史价判断**必须**先调 `convert` 获取到手价，再调 `history-price` 获取历史数据
- 比价和历史价接口可选 `--shopType` 参数指定平台，枚举值：`C`=淘宝、`B`=天猫、`P`=拼多多、`J`=京东、`DY`=抖音、`V`=唯品会、`MT`=美团

## 调用示例

```bash
# 转链
bash {baseDir}/scripts/run.sh call convert --tpwd "https://u.jd.com/xxx"

# 跨平台比价
bash {baseDir}/scripts/run.sh call compare-price --productIdentifier "https://u.jd.com/xxx"

# 历史价格（先转链拿到手价，再查历史）
bash {baseDir}/scripts/run.sh call convert --tpwd "https://u.jd.com/xxx"
bash {baseDir}/scripts/run.sh call history-price --productIdentifier "https://u.jd.com/xxx"

# 全套（用户只发链接或问值不值得买时）
bash {baseDir}/scripts/run.sh call convert --tpwd "https://u.jd.com/xxx"
bash {baseDir}/scripts/run.sh call compare-price --productIdentifier "https://u.jd.com/xxx"
bash {baseDir}/scripts/run.sh call history-price --productIdentifier "https://u.jd.com/xxx"
```

## 输出呈现规范

所有输出使用 Markdown 格式，面向飞书/微信等 IM 场景渲染。

**IMPORTANT — 流式输出原则**：用户在聊天界面等待回复。每多等 1 秒，体验就差一分。因此：
1. **收到用户消息后，先输出文字再调接口**——不要让用户面对空白等待
2. **每个接口返回后，立刻输出对应卡片**——不要攒齐所有数据再统一渲染
3. 你的回复是一条不断追加的消息，不是一次性完成的文档

**字段规则**：

- `itemPicUrl` 不存在 → 省略图片行

### 历史价 → 购买建议 + 价格走势

输出分三部分：购买建议 → 价格对比表 → 近期走势表。

**数据准备：**
- 到手价 = convert 接口的 `finalPrice`
- 历史最低价 = history-price 接口的 `historyLowestPrice`
- 30天最低价 = `lowest30DaysPrice`
- 日常价 = `usualPrice`
- 省的金额 = 历史最低价 - 到手价（仅当到手价 < 历史最低价时有意义）
- 价格波动幅度 = (日常价 - 历史最低价) ÷ 日常价
- 价格位置 = (到手价 - 历史最低价) ÷ (日常价 - 历史最低价)（仅当日常价 ≠ 历史最低价时计算）

**购买建议规则（严格按优先级判断，命中即停）：**

| 优先级 | 情景 | 判定条件 | 输出 |
|--------|------|----------|------|
| 1 | 破历史低（有感） | 到手价 < 历史最低价，且（省 ≥ ¥20 或省 ≥ 5%） | 🟢 **比历史最低还便宜 ¥X，强烈推荐入手！** |
| 2 | 破历史低（微小） | 到手价 < 历史最低价，但省 < ¥20 且 < 5% | 🟢 **当前到手价略低于历史最低，适合入手** |
| 3 | 价格恒定 | 价格波动幅度 < 3% **且** 到手价在日常价 ±3% 以内 | 🟡 **该商品长期价格稳定在 ¥X 左右，不存在更低时机，需要就买** |
| 4 | 历史最低附近 | 到手价 ≤ 历史最低价 × 1.03 | 🟢 **到手价在历史最低附近，适合入手** |
| 5 | 低位 | 价格位置 ≤ 0.3 | 🟢 **接近历史低点，适合入手** |
| 6 | 中位 | 价格位置 0.3 ~ 0.7 | 🟡 **价格适中，{大促提示}** |
| 7 | 高位 | 价格位置 > 0.7 | 🔴 **高于日常水平，建议等降价** |

**大促提示（用于中位情景）：**

大促日历：年货节(1月)、38节(3月)、618(6月)、双11(11月)、双12(12月)

| 条件 | 提示 |
|------|------|
| 距下一个大促 ≤ 30天 | 距{大促名}还有{N}天，不急可以等 |
| 距下一个大促 > 30天 | 短期内没有大促，按需购买即可 |

**波动补充语（追加在建议之后，仅在不矛盾时使用）：**

用 curveData 最近 3 个数据点判断（不足 3 个则不输出波动判断）：

| 波动特征 | 判定 | 补充语 | 限制 |
|----------|------|--------|------|
| 持续下降 | 最近3点递减 | 📉 近期价格在下降，可以再观望几天 | 仅在🟡或🔴时追加 |
| 持续上涨 | 最近3点递增 | 📈 近期价格在回升，再等可能更贵 | 仅在🟡或🔴时追加 |
| 反复波动 | 价格来回跳动 | 🔄 价格周期性波动，逢低入手即可 | 仅在🟡时追加 |
| 稳定 | 所有数据点价差 < 3% | 不追加（已在主建议体现） | — |

**券/国补独立提示（满足条件就追加，不冲突）：**
- `couponAmount` ≥ 原价 10% → ⏰ **含 ¥X 大额优惠券，可能限时限量**
- `countrySubsidyFlag` 不是"无国补" → 🏷️ **该商品支持国家补贴，叠加后更划算**

**输出模板：**

```markdown
### 💰 购买建议：{建议结论}

{建议详情，含波动补充语和券/国补提示}

---

| 指标 | 价格 | 备注 |
|------|------|------|
| 到手价 | ¥{finalPrice} | 含券 |
| 历史最低价 | ¥{historyLowestPrice} | {historyLowestDate} |
| 30天最低价 | ¥{lowest30DaysPrice} | — |
| 日常价 | ¥{usualPrice} | — |

**近期价格走势：**

| 日期 | 价格 | 趋势 |
|------|------|------|
| {date} | ¥{price} | {🟢降/🔴涨/➡️平} |
| ... | ... | ... |
```

趋势列：与前一个数据点比较，下降标🟢，上涨标🔴，持平标➡️。第一个数据点标➡️。

### 比价 → 推荐 + 对比表

**比价建议规则：**

| 情景 | 判定条件 | 输出 |
|------|----------|------|
| 仅1家 | totalCount = 1 | 仅找到一家在售，建议结合历史价格判断是否值得入手 |
| 一家明显低 | 最低与次低差价 > 5% **且** > ¥10 | 🟢 **推荐 {平台}·{店铺}，比第二低便宜 ¥X** |
| 价格接近 | 差价 ≤ 5% 或 ≤ ¥10 | 🟡 **多家价格接近（¥X ~ ¥Y），建议选常用且售后方便的平台** |

同价时优先推荐：badge = "全网最低" 或 choice = 1 的店铺。

```markdown
| 平台 | 店铺 | 价格 | 标签 |
|------|------|------|------|
| {shopType} | {shopName} | **¥{price}** | {badge或"—"} |
| ... | ... | ... | ... |
```

### 全套流程（用户只发链接或问值不值得买时）

**IMPORTANT：全套流程的每一步都是独立的"输出文字 → 调接口 → 输出卡片"循环。你必须在调用每个接口之前就先输出文字给用户，在每个接口返回后立刻输出对应卡片。禁止等所有接口都调完再统一输出。**

下面是你收到用户链接后必须严格遵循的完整行为序列：

**→ 立刻输出**：`正在查询优惠信息...`（这是你的第一句话，让用户知道你在工作）
**→ 调用** `convert` 接口
**→ 拿到结果后立刻输出**：按「购买卡片」模板渲染，输出 `---` 分隔线

**→ 立刻输出**：`正在查询历史价格...`
**→ 调用** `history-price` 接口
**→ 拿到结果后立刻输出**：按「购买建议 + 价格走势」模板渲染，输出 `---` 分隔线

**→ 立刻输出**：`正在比价...`
**→ 调用** `compare-price` 接口
**→ 拿到结果后立刻输出**：按「比价对比表」模板渲染

某步接口失败则告知用户该步数据暂不可用，继续下一步。

## 错误处理

| 现象 | 处理 |
|------|------|
| `missing_parameter` | 提示用户提供商品链接或淘口令 |
| `api_unavailable` | 告知用户服务暂时不可用，建议稍后重试 |
| `errorMessage: "未找到相关商品"` | 提示用户检查链接是否正确 |
| `topLowestItems: null` 或 `historyLowestPrice: null` | 告知用户该商品暂无对应数据 |
| `ModuleNotFoundError: No module named 'yaml'` | 提示用户执行 `pip3 install pyyaml` 安装依赖 |

## 环境依赖

- **Python 3**：脚本运行需要 Python 3
- **pyyaml**：读取配置文件需要，安装命令：`pip3 install pyyaml`
- **curl**：HTTP 请求
- API 地址和密钥在 `{baseDir}/scripts/_env.yaml`，默认已配好
